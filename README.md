# TableCRM Mobile Order


[Открыть приложение](https://table-crm-mobile-order-1.vercel.app/)

Мобильное webapp-приложение для оформления заказа в TableCRM.

Проект выполнен как **mobile-first форма создания продажи** в формате одного экрана: пользователь вводит токен, загружает справочники, находит клиента, выбирает товары, формирует корзину и отправляет заказ в TableCRM.

---

## Что было нужно по тестовому

Реализовать мобильную форму оформления заказа для `tablecrm.com`, близкую по смыслу к модальному окну создания продажи в основной системе.

Основные требования:

- ввод токена для авторизации в конкретной кассе
- поиск клиента по телефону
- выбор:
  - счетов
  - организаций
  - складов
  - типов цен
- выбор товаров
- действия:
  - **создать продажу**
  - **создать и провести**
- UI на базе **shadcn/ui**
- стек: **Next.js / TypeScript / Tailwind / shadcn**

---

## Что реализовано

### Основной пользовательский flow

- один мобильный экран без лишней многостраничности
- подключение к TableCRM по токену
- загрузка справочников после подключения
- поиск клиента по телефону
- выбор организации, склада, счета и типа цены
- поиск товаров по названию
- добавление товаров в корзину
- редактирование товара в корзине:
  - количество
  - цена
  - скидка
- расчет итоговой суммы на клиенте
- отправка заказа в TableCRM
- сценарий **«создать и провести»** через второй запрос на смену статуса

### UX и техническая часть

- mobile-first интерфейс
- loading / empty / error states
- toast-уведомления для действий и ошибок
- клиентская валидация обязательных полей
- выделенная функция сборки payload: `buildSalePayload`
- отделение UI, API-слоя и payload-логики

## Архитектурная идея

Задача решена как **один основной экран**, потому что для тестового важнее быстрый и понятный пользовательский flow, чем сложная многостраничная навигация.

Логика разделена на 3 уровня:

- **UI-компоненты** — форма, секции, поля, корзина
- **API-слой** — запросы к TableCRM и обработка ответов
- **бизнес-логика** — подготовка payload и вычисление итогов

Такой подход упрощает поддержку, тестирование и дальнейшее расширение формы.

## Бизнес-логика payload

Для MVP при создании заказа используются следующие значения:

- `operation: "Заказ"`
- `tax_included: true`
- `tax_active: true`
- `priority: 0`
- `paid_lt: 0`
- `status: false`
- `settings: {}`
- `dated` = текущий Unix timestamp в секундах

Если у выбранного клиента есть карта лояльности, в payload добавляется:

```json
"loyality_card_id": 123
```

Если карты нет, поле не отправляется.

---

## Submit-логика

### Создать продажу

Используется запрос:

```http
POST /api/v1/docs_sales/?token=...&generate_out=false
```

Body отправляется как массив из одного документа.

### Создать и провести

Используется двухшаговый сценарий:

1. создание продажи
2. смена статуса созданного документа

Запросы:

```http
POST /api/v1/docs_sales/?token=...&generate_out=false
PATCH /api/v1/docs_sales/{id}/status?token=...
```

Body второго запроса:

```json
{
  "status": "processed"
}
```

## Валидация перед отправкой

Кнопки отправки недоступны, если:

- не введен токен
- не выбран клиент
- не выбрана организация
- не выбран склад
- не выбрана касса
- корзина пуста

Дополнительно при попытке действия без токена показывается верхняя ошибка, чтобы пользователь сразу понял причину.

## Ограничения и допущения

Ниже только те ограничения, которые действительно влияют на проверку задания.

- Для полного end-to-end сценария нужен валидный токен TableCRM.
- Сценарий **«создать продажу»** является основным MVP-flow и наиболее надежен для проверки.
- Для сценария **«создать и провести»** необходимо корректно получить `id` созданной продажи из ответа API.
- Ответ `POST /docs_sales/` описан в документации не полностью, поэтому извлечение `id` сделано через безопасный runtime fallback.
- Приложение работает напрямую с внешним API TableCRM без собственного backend proxy, поэтому зависит от доступности API, CORS и валидности токена.

## Почему это решение выглядит как законченное тестовое

В этом проекте я старался закрыть не только визуальную часть формы, но и полный пользовательский путь:

- подключение
- загрузка справочников
- выбор сущностей
- работа с корзиной
- сборка валидного payload
- отправка заказа
- обработка ошибок и состояний интерфейса

То есть решение ориентировано не на «макет формы», а на рабочий MVP-flow, максимально близкий к реальному использованию.
